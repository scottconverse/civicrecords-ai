## Findings

### HIGH — Wrong Data Displayed

**Finding 1: Frontend version hardcoded, mismatches backend**
- **Where:** Dashboard header shows "CivicRecords AI v1.0.1", footer shows "CivicRecords AI v1.0.0 - Apache 2.0"
- **Expected:** v1.1.0 (matching APP_VERSION in backend config.py)
- **Root Cause:** Frontend has hardcoded version strings. The dashboard header likely reads from `/health` API (which returns v1.1.0 if the container has current code) OR is hardcoded in React. The footer is definitely hardcoded.
- **Impact:** Any evaluator looking at the system sees the wrong version.
- **Fix:** Update version strings in frontend source, or make the footer read from the API.

**Finding 2: Only 1 exemption rule in production database**
- **Where:** Exemptions page shows "ACTIVE RULES 1/1" — only one CO rule visible
- **Expected:** 180 rules across 51 jurisdictions (50 states + DC)
- **Root Cause:** The `seed_rules.py` script was updated but never run against the production Docker database. Rules exist in the test DB only.
- **Impact:** Exemption scanning will only detect one rule pattern. The entire 50-state coverage is non-functional in production.
- **Fix:** Run `docker compose exec api python -m scripts.seed_rules` against the production database.

### MEDIUM — Missing UI Feature

**Finding 3: No department column or assignment on Users page**
- **Where:** Users page (/users)
- **Expected:** Department column in the user table, department selector when creating/editing users
- **Current:** Table shows Name, Email, Role, Status, Last Active — no department info
- **Impact:** Admins cannot see or assign user departments from the UI. Department management only possible via API.
- **Fix:** Add department column to users table, add department dropdown to Create User form.

**Finding 4: Status badges lack distinct icons per status**
- **Where:** Requests list page (/requests), status column
- **Expected:** Canonical spec Section 6 requires distinct icons per status: Inbox (received), MessageCircle (clarification), UserCheck (assigned), Search (searching), Eye (in review), CheckCircle (ready for release), FileText (drafted), ShieldCheck (approved), Send (fulfilled), Archive (closed). "Every badge includes an icon — never color-only."
- **Current:** Badges show a generic colored circle + text. Same circle shape for all statuses.
- **Fix:** Import Lucide icons and map each status to its specified icon.

**Finding 5: Liaison and public roles not in frontend Create User form**
- **Where:** Users page → Create User button (presumed)
- **Expected:** Role dropdown should include all 6 roles: admin, staff, reviewer, read_only, liaison, public
- **Current:** Backend enum has all 6. Frontend role dropdown likely only shows the original 4.
- **Fix:** Update the role dropdown options in the frontend Create User component.

### LOW — Cosmetic / Cleanup

**Finding 6: XSS and SQL injection test data in production database**
- **Where:** Requests list page
- **Visible:** `<script>alert(1)</script>` as requester names, `<img onerror=alert(1) src=x>` in descriptions, `'; DROP TABLE users; --` as requester name
- **Positive:** All properly escaped — React prevents execution. No security vulnerability.
- **Impact:** Unprofessional appearance for any demo or evaluation.
- **Fix:** Clean up test data from production database.

**Finding 7: Footer version "v1.0.0" on every page**
- **Where:** Bottom of every page
- **Current:** "CivicRecords AI v1.0.0 - Apache 2.0"
- **Fix:** Part of Finding 1 — update footer version string.

**Finding 8: Sources page integration cards show "Coming in v1.1" / "Coming in v2.0"**
- **Where:** Sources page (/sources), Integration cards
- **Current:** Database (ODBC) says "Coming in v1.1", API Endpoint says "Coming in v2.0"
- **Issue:** These version labels use the old version numbering that doesn't match the canonical spec's phasing. Minor since this is a placeholder UI element.
- **Fix:** Low priority — update when connector work actually starts.
