CivicRecords AI — Project Decomposition

How should we break this into buildable sub-projects?

The System Has 5 Major Subsystems

Each could be its own spec → plan → build cycle. The question is sequencing.

Architecture Overview
1. FOUNDATION
Docker + PostgreSQL/pgvector + FastAPI skeleton + Auth + Audit logging
Must be first. Everything depends on this.
2. INGESTION PIPELINE
Two-track: parsers (DOCX, CSV, email, text) + Gemma 4 multimodal (scans, images, OCR)
Feeds the knowledge base. No search without this.
3. RAG SEARCH ENGINE
Ollama + pgvector + hybrid search + source attribution + React search UI
The core value prop. This is what staff use every day.
4. REQUEST TRACKING
Case management, deadlines, document association, basic request logging
Turns search results into managed workflow.
5. EXEMPTION DETECTION & COMPLIANCE
Rules engine (PII regex, statutory phrases) + LLM suggestion layer + exemption flag workflow + compliance templates + auditability dashboard
Builds on top of search + request tracking. Phase 2 territory.

Proposed Sequencing

A

Linear: 1 → 2 → 3 → 4 → 5

Foundation first, then ingestion, then search, then tracking, then exemptions.

Each sub-project fully complete before the next starts. Simplest to manage. MVP (usable search) arrives after sub-projects 1-3 are done. Slower to first value but each piece is solid.

Pros

  • Clean dependencies
  • Each piece tested in isolation
  • Easiest for 1 person

Cons

  • Longest time to first usable product
  • No workflow until sub-project 4
B

Vertical Slice: 1 → (2+3+4 minimal) → iterate

Foundation, then a thin slice through all three MVP subsystems at once.

Build the minimum of each: ingest PDFs only, basic search, simple request form. Get something end-to-end working fast, then deepen each subsystem iteratively. MVP arrives sooner but each piece starts thin.

Pros

  • Fastest to something usable
  • End-to-end validation early
  • Can demo/test with real users sooner

Cons

  • More moving parts at once
  • Risk of thin features that need rework
  • Harder to track progress
C

Hybrid: 1 → 2 → 3 → (3+4 integrated) → 5

Foundation, ingestion, and search as complete sub-projects. Then integrate request tracking into the working search UI. Exemptions last.

You get a fully working search tool after sub-project 3 (already useful to a clerk). Then add request tracking as an enhancement to the search experience, not a separate system. Exemption detection builds on both.

Pros

  • Search is usable standalone after sub-project 3
  • Request tracking builds on proven search UX
  • Natural progression of value
  • Each spec is focused

Cons

  • Slightly slower than B to full MVP
  • Request tracking design might change search UI

My recommendation is C — it gives you a usable product after sub-project 3, and each piece is solid before the next builds on it.
